# 多智能体协作基准测试：一个框架无关的智能体AI系统对比工具

**Junze Li — DAI 2026 工业赛道**

---

## 摘要

多智能体AI框架——CrewAI、AutoGen、LangGraph等——的快速涌现催生了对标准化评估方法的迫切需求。虽然每个框架都为智能体协作提供了独特的抽象，但现有的评估工具仍然碎片化——大多数基准测试针对单模型能力而非多智能体协作质量，而新兴的框架对比工具仍处于早期阶段。我们提出 **Multi-Agent Benchmark (MAB)**，一个开源的、框架无关的工具包，用于在标准化协作密集型任务上评估多智能体系统。MAB 提供：(1) 统一的适配器接口，将任何智能体框架包装在通用API之后；(2) 精心策划的9个基准任务套件，涵盖问答、代码生成、数据分析、研究综述和规划；(3) 自动插桩采集token消耗、延迟、成本、交互模式和编排开销；(4) 交互式可视化仪表盘用于对比分析。我们通过108次实验运行评估MAB，跨越12种配置（覆盖DeepSeek和OpenAI），包含一个真实的CrewAI实现以验证我们发现的有效性。四个发现浮现：(1) 多步协作普遍提升了所有模型的性能（平均得分+0.05--0.16），但增益与单智能体能力成反比——越弱的模型受益越多；(2) 简单的结构分解在所有测试模型上都优于复杂的基于角色的协作——更丰富的提示词只增加了token开销而不提升质量；(3) 最佳配置（DeepSeek V4 Flash + 多步协作）以$0.016的总成本实现100%成功率，比达到同等成功率的GPT-4o便宜11.6倍；(4) 协作可以增强推理但无法创造知识——RES-2上的厂商级分裂表明，缺失的事实无法通过增加更多智能体来弥补。我们的工具包使从业者能够在模型选择和协作架构两方面做出数据驱动的决策。

## 1. 引言

AI智能体开发的格局在过去两年经历了巨变。2024--2026年间，我们目睹了大量多智能体框架的出现：CrewAI、微软的AutoGen、LangChain的LangGraph等。这些框架使开发者能够将多个LLM驱动的智能体组合成协作系统，能够分解复杂任务、分配专门角色，并——原则上——通过结构化协作产生更高质量的输出。

然而，一个关键的空白仍然存在：**标准化的跨框架评估仍处于萌芽阶段**。虽然最近的努力如Arena和MAFBench已经开始处理框架对比问题，但它们主要关注在固定模型下框架级别的编排差异。从业者面临的问题是——"我应该为我的用例选择哪个框架*和*哪个模型？"——却缺乏足够的实证指导。模型选择与协作策略之间的交互效应仍然未被充分探索。如今，框架选择往往由文档质量和社区热度引导，而非经过实测的、特定于任务的性能表现。

这个问题对**工业界从业者**尤为尖锐。与可以在狭义强化学习基准上评估的学术研究者不同，工业团队需要理解：

- **任务成功**：多智能体系统真的能解决问题吗？
- **成本效率**：它消耗了多少API调用和token？
- **时间开销**：智能体协调的延迟代价是多少？
- **鲁棒性**：在更困难的任务或边界情况下性能会下降吗？

我们提出**Multi-Agent Benchmark (MAB)**，一个框架无关的工具包，旨在填补这一空白。MAB将每个智能体配置——无论是单次LLM调用还是多步协作流水线——视为统一接口背后的"黑箱"，在相同任务上运行所有配置，并产生可比较的指标。该工具包具有：

- **可扩展**：只需实现一个适配器类即可添加新框架。
- **可复现**：任务具有确定性输入和自动化评估。
- **实用**：指标聚焦于生产环境中的关键因素——成功率、token成本、延迟和编排开销。
- **开源**：以MIT许可证发布，供社区采纳和扩展。

本文做出以下贡献：

1. **一个框架无关的基准测试工具包**，具有统一适配器接口、插桩式指标采集和交互式可视化仪表盘。
2. **一个精心策划的9个协作密集型任务套件**，涵盖4个类别，具有自动化评估。
3. **一个12种配置的实证研究**（108次运行），揭示了普遍协作红利、能力天花板效应，以及简单的三步流水线在V4 Flash上优于基于角色的适配和真实CrewAI的发现。

## 2. 相关工作

### 2.1 多智能体框架

过去两年见证了多智能体框架的爆发。CrewAI将智能体组织成具有定义角色、目标和顺序或层级任务执行的"团队"。AutoGen将智能体交互建模为结构化对话，通过群聊管理器编排多智能体对话。LangGraph将智能体工作流表示为有向状态图，提供显式控制流。每个框架体现了不同的设计哲学：基于角色的分解（CrewAI）、对话驱动协作（AutoGen）和基于图的编排（LangGraph）。最近的工作已开始对比这些框架，但框架选择、模型选择和任务类型之间的交互仍然未被充分探索。

### 2.2 LLM评估基准

现有的LLM基准——MMLU、HumanEval、Big-Bench、GAIA——评估的是单模型能力。它们不衡量多智能体协作质量。SWE-Bench在软件工程任务上评估智能体，但绑定于特定领域且不比较框架。AgentBench在多样化环境中评估单智能体，但不涉及多智能体协调。

### 2.3 AgentOps与可观测性

最近关于AgentOps的工作确立了可观测性在智能体系统中的重要性。我们的工具包构建在这些洞察之上，将指标采集作为一等公民——每次LLM调用都被插桩，每次交互都被追踪，开销被量化。与商业AgentOps平台（LangSmith、AgentOps.ai、Braintrust）不同，MAB专为跨框架对比而非生产监控而设计。

### 2.4 多智能体评估

最近几项工作已经开始处理框架对比的空白。Arena在三个自定义场景上比较了六个智能体框架（Claude Agent SDK、LangChain、LangGraph、AWS Strands、CrewAI、Google ADK），固定底层模型以隔离框架编排效应。MAFBench为多智能体框架提出了架构分类法，并提供了统一的评估套件。BenchAgent研究了在规范化协议下增加智能体是否能提升性能，发现大多数多智能体系统并未超过单智能体基线。MASEval提供了一个PyPI包，用于针对现有基准（如GAIA和AgentBench）进行多智能体系统评估。

MAB与这些工作有**三个关键差异**。首先，我们**联合变化模型层级和协作策略**，揭示了先前工作——通常固定其中一个维度——无法观察到的模型-协作交互效应。其次，我们将**成本-性能帕累托分析**作为一等的评估维度，反映了工业采纳同时取决于质量和成本的现实。第三，我们的**轻量级、可扩展的适配器架构**（每个框架约100行代码）和交互式Streamlit仪表盘降低了从业者想要评测自己特定任务的门槛，而非依赖预打包的基准套件。

## 3. 系统设计

### 3.1 架构概览

MAB遵循五层模块化架构：

```
可视化仪表盘 (Streamlit)
        ↓
基准测试运行器 & 报告生成器
        ↓
框架适配器 (Single-Agent, CrewAI, AutoGen, LangGraph)
        ↓
任务套件 (9个任务, 4个类别)
        ↓
插桩式LLM层 (指标采集)
```

### 3.2 插桩式LLM层

跨框架对比的基础是一致的指标采集。MAB的`InstrumentedLLM`类包装LLM客户端（支持Anthropic、OpenAI和DeepSeek后端），并透明地记录每次API调用：

- 提示词和补全的token数
- 调用持续时间（毫秒）
- 智能体身份（哪个智能体发起了调用）
- 调用目的（推理、工具调用、摘要等）

这种方法确保无论哪个框架编排智能体，指标采集完全一致。所有框架共享相同的底层LLM后端，使用相同的超参数（temperature、最大token数、模型版本），消除了混杂变量。

### 3.3 框架适配器

每个适配器实现`BaseAdapter`接口：

```python
class BaseAdapter(ABC):
    def solve(self, task_input: TaskInput) -> str: ...
    def run(self, task_input) -> tuple[str, RunMetrics]: ...
    def agent_names(self) -> list[str]: ...
```

适配器负责：(1) 将基准任务翻译为框架特定的智能体配置，(2) 执行智能体，(3) 返回最终输出。这种设计使添加新框架变得简单——每个框架约100行适配器代码。

### 3.4 任务设计

MAB中的每个任务由三个组件定义：

- **InputGenerator**：产生确定性的任务输入，包含问题描述和结构化数据。
- **Evaluator**：对输出字符串在0.0--1.0的范围内评分，使用关键词匹配、测试用例执行（代码任务）或结构化答案验证。
- **Metadata**：类别、难度、推荐最少智能体数、预期时长和标签。

任务被设计为**本质上需要协作**——它们需要多个推理步骤、多样化知识或不同的技能组合，这些都能从智能体专门化中受益。

### 3.5 指标

对于每个（任务，适配器）对，MAB采集以下核心指标：

| 指标 | 描述 |
|------|------|
| Score | 任务成功度（0.0--1.0，自动化评估） |
| Total Tokens | 提示词+补全token之和 |
| Prompt Tokens | 输入上下文消耗的token |
| Completion Tokens | 输出生成的token |
| Total Time (ms) | 运行的端到端耗时 |
| Agent Interactions | 智能体间消息交换次数 |
| Estimated Cost (USD) | 基于模型特定定价 |
| Orchestration Overhead | LLM调用之外花费的时间 |
| Efficiency Ratio | 补全token / 总token |

## 4. 基准任务套件

MAB包含9个任务，跨越4个类别：

| ID | 任务 | 类别 | 难度 |
|----|------|------|------|
| MHQ-1 | AlphaGo总部所在国的官方语言 | 多跳问答 | 中等 |
| MHQ-2 | 第一个登月者的出生年份 | 多跳问答 | 中等 |
| MHQ-3 | 爱因斯坦同名元素的化学符号 | 多跳问答 | 中等 |
| CODE-1 | 回文判断函数 | 代码生成 | 中等 |
| CODE-2 | 斐波那契数列函数 | 代码生成 | 中等 |
| DATA-1 | 销售数据分析报告 | 数据分析 | 中等 |
| RES-1 | Transformer vs Mamba | 研究综述 | 困难 |
| RES-2 | 推测解码 | 研究综述 | 困难 |
| PLAN-1 | 会议组织策划 | 规划类 | 中等 |

### 4.1 多跳问答（MHQ-1,2,3）

这些任务需要串联多个事实：例如，"创造了AlphaGo的公司总部所在国家的官方语言是什么？"正确回答需要识别AlphaGo的创造者（DeepMind）、其位置（英国伦敦）和官方语言（英语）。单智能体可能跳过步骤或幻觉中间事实；多智能体系统可以由一个智能体验证推理链中的每个环节。

### 4.2 代码生成+审查（CODE-1,2）

编码任务不仅需要生成代码，还需要生成能通过测试用例的**正确**代码。多智能体设置模拟真实的开发工作流：一个智能体编写初始代码，另一个审查其正确性和边界情况，第三个生成最终版本。评估运行实际的Python测试用例。

### 4.3 数据分析（DATA-1）

给定表格化的销售数据，智能体必须计算总计、识别趋势并生成结构化报告。这需要计算准确性、领域推理和清晰的沟通——这些技能从角色专门化中受益。

### 4.4 研究综述（RES-1,2）

这些任务需要深厚的领域知识（transformer架构、推测解码）和构建全面解释的能力。多个智能体可以并行研究不同方面并综合发现。

### 4.5 规划类（PLAN-1）

开放式规划任务测试智能体系统系统性组织信息、思考约束条件和生成带有定量估计的可执行计划的能力。

## 5. 实验结果

### 5.1 实验设置

我们评估了12种配置，跨越4个模型类型、2个厂商，包括一个真实的CrewAI实现以验证我们基于适配器的发现：

- **DeepSeek V4 Flash**（285B总参数量/13B激活参数，$0.14/$0.28每百万token）和**DeepSeek V4 Pro**（1.6T总参数量/49B激活参数，$0.435/$0.87）。
- **GPT-4o Mini**和**GPT-4o**（OpenAI，通过API代理）。成本估算使用官方定价；实际代理成本可能不同。

对于每个模型，我们测试：(1) **单智能体**——一次LLM调用完成整个任务；(2) **多步协作**——三阶段流水线（研究员→分析师→撰稿人）；(3) **基于角色的协作**——相同流水线增加了CrewAI风格的角色定义、背景故事、委托和expected_output规范，在V4 Flash、GPT-4o Mini和GPT-4o上测试；(4) **真实CrewAI**（v1.14）——实际的CrewAI库，具有原生DeepSeek集成，在V4 Flash上测试以验证基于适配器的发现。所有实验使用temperature=0.0。每组运行一次。总计：108次运行（9个任务 × 12种配置）。API总成本：$0.60。

### 5.2 总体表现

以下三个发现引人注目：

1. **DeepSeek V4 Flash + 多步协作取得了最佳总体结果**（0.91，100%成功率），超越所有其他配置，包括那些使用显著更昂贵模型的配置（V4 Pro、GPT-4o）。
2. **多步协作普遍提升了性能。**每个模型都从研究员-分析师-撰稿人流水线中受益，平均得分增益在+0.05到+0.16之间。
3. **最佳多步配置的成功率收敛到100%**（V4 Flash、GPT-4o），表明结构化协作可以弥补较低的单次通过能力。

**综合结果表：**

|------|------|--------|-------|---------|------|

| **V4 Flash Multi** | **0.91** | **100%** | 69,584 | 41.5 | $0.016 |
| V4 Pro Single | 0.80 | 89% | 13,007 | 27.1 | $0.011 |
| V4 Pro Multi | 0.85 | 89% | 89,386 | 164.2 | $0.064 |
| GPT-4o Mini Single | 0.67 | 67% | 6,256 | 8.3 | $0.003 |
| GPT-4o Mini Multi | 0.83 | 78% | 37,254 | 28.1 | $0.014 |
| GPT-4o Single | 0.79 | 89% | 5,972 | 5.5 | $0.047 |
| GPT-4o Multi | 0.85 | 100% | 30,496 | 19.7 | $0.184 |

### 5.3 普遍协作红利

最重要的发现是跨厂商的普适性：**协作红利在所有四个模型类型中都成立，但其幅度与单智能体能力成反比**。GPT-4o Mini，最弱的单智能体（0.67），从协作中获益最多（+0.16）。GPT-4o和V4 Pro，最强的单智能体（0.79--0.80），获益最少（+0.05--0.06）。V4 Flash占据了一个最佳平衡点——强大的单智能体表现（0.79）仍能从协作中获得显著收益（+0.12）。

这种模式暗示了一个**能力天花板效应**：当单智能体表现接近任务的难度天花板时，额外的协作阶段提供递减的回报。最弱的模型有最大的空间从结构化多步处理中受益。

**基于角色的 vs. 简单多步 vs. 真实CrewAI。**在V4 Flash上，我们的简单多步适配器得分0.91，真实CrewAI（v1.14）得分0.89，基于角色的适配器得分0.84。真实CrewAI的原生编排优于我们的角色近似，但仍然不及最简单的三步流水线——尽管其设计中投入了更多的工程努力。基于角色的适配器在所有三个测试模型上都不及简单的多步适配器：V4 Flash（0.84 vs. 0.91）、GPT-4o Mini（0.70 vs. 0.83）和GPT-4o（0.76 vs. 0.85）。最弱模型的差距最大（GPT-4o Mini：-0.13），表明复杂的角色扮演提示词可能使能力较弱的模型更加困惑。这些结果与Arena的发现"提示词而非代码驱动智能体性能"一致：**结构化分解比角色扮演的精细程度或框架的复杂性更重要**，这一原则在模型层级和厂商之间都成立。

### 5.4 Token经济学

多步配置消耗的token是其单智能体对应配置的5--7倍，在所有模型中都如此。token乘数惊人地一致（5.1--6.9×），无论厂商或模型层级——这是三阶段流水线的结构性属性，而非模型特定的行为。

**最佳性价比**：V4 Flash Multi以$0.016的总成本实现100%成功率——比达到同等成功率的GPT-4o Multi（$0.184）便宜11.6倍。**最差性价比**：V4 Pro Multi（$0.064，89%成功率）在帕累托意义上被V4 Flash Multi全面压制。

### 5.5 编排开销

V4 Pro在多步模式下表现出极高的开销（164秒/任务），由较慢的逐token生成在三阶段序列中被放大。GPT-4o是最快的单智能体（5.5秒），在多步模式下保持合理速度（19.7秒），表明逐token延迟而非流水线深度是端到端延迟的主导因素。

### 5.6 任务级别发现

**代码生成从协作中获益最多。**CODE-1（回文判断）展示了最显著的提升：GPT-4o Mini作为单智能体得分为0.00，但多步协作后达到0.92——分析师阶段捕获了单次通过无法发现的实现错误。CODE-2（斐波那契）展示了模型依赖的结果：V4 Flash提升（+0.36）和GPT-4o Mini提升（+0.18），但GPT-4o无变化，V4 Pro轻微退化（-0.09），突显了协作有时可能过度纠正或稀释一个可行的解决方案。

**多跳问答展示了模型特定的模式。**MHQ-1（AlphaGo→英国→英语）是最具区分度的任务。单智能体得分在0.20--0.60之间，而多步将DeepSeek Flash提升到1.00。GPT-4o模型无论策略如何都停留在0.40--0.60，表明这是一个知识缺口而非推理失败。

**大多数知识任务接近天花板。**MHQ-2、PLAN-1和RES-1被几乎所有配置满分解决。MHQ-3在12种配置中有11种满分（仅GPT-4o Mini Single以0.70落败）。相比之下，Research-2（推测解码）揭示了厂商级分裂：DeepSeek模型取得满分，而GPT模型无论策略如何都停留在0.49--0.63，表明预训练知识覆盖的差异而非推理能力。

## 6. 讨论

### 6.1 解读结果

我们数据中的三个模式值得更深入的讨论。

**为什么协作红利因模型层级而异？**能力天花板效应——较弱的模型从协作中获益更多——有一个自然的解释：多步流水线对那些单次通过无法解决的任务最有帮助。GPT-4o Mini在CODE-1上的显著提升（0.00→0.92）发生在其单次通过完全失败的情况下，留下了最大的改进空间。相反，V4 Pro单Agent已经得分0.80，多步协作仅将其提升至0.85——远低于实测最佳成绩（0.91）。其最佳成绩与全局最佳之间的差距，反映的是协作无法单独弥补的内在能力上限。这暗示了一个实用的启发式规则：**首先对你的单智能体性能进行基准测试；如果低于0.70，协作可能值得它的成本；高于0.85，可能不值得。**

**RES-2上的厂商分裂告诉我们什么？**所有DeepSeek模型在推测解码上得分为1.0，而所有GPT模型无论协作策略如何都停留在0.49--0.63——这表明是预训练数据覆盖的缺口，而非推理缺陷。这有一个重要的含义：**协作增强推理但不能创造知识。**当一个模型缺乏底层事实时，添加更多具有相同知识缺口的智能体没有帮助。从业者应在投资多智能体架构之前诊断任务失败是源于知识缺口还是推理失败。

**三阶段流水线是最优的吗？**我们的实验将研究员-分析师-撰稿人流水线作为固定设计，但最优的协作结构可能取决于任务类型。CODE-1主要受益于分析师阶段（捕获错误），而MHQ-1受益于研究员阶段（分解推理链）。撰稿人阶段贡献的输出质量提升可能未被我们粗粒度的评分捕获。未来的工作应探索**自适应流水线构建**——基于任务特征选择包含哪些阶段——以避免在只有一个阶段增加价值时支付完整的5--7倍token乘数。

### 6.2 局限性

应注意以下几点局限性：

- **任务天花板效应**：9个任务中只有3个（MHQ-2、PLAN-1、RES-1）被几乎所有配置满分解决，但另有几个（MHQ-3、DATA-1）展示了接近天花板的分数，配置间区分度极小。未来版本需要更难的任务以在更高能力水平上更好地区分配置。
- **任务多样性**：我们的9个任务虽然跨越4个类别，但未覆盖所有重要的多智能体用例（例如协商、辩论、工具链、带状态跟踪的长期规划）。
- **评估粒度**：基于关键词匹配和测试用例的自动评分是粗粒度的。两个得分相同的输出在质量上可能有显著差异。人工评估将提供更精细的区分，特别是对于开放式任务。
- **协作模式 vs. 框架**：我们的适配器捕获了基本的协作模式（简单流水线、基于角色），但不反映生产系统（如CrewAI或LangGraph）可能采用的框架特定优化（并行执行、缓存、动态路由）。然而，简单的多步流水线在V4 Flash上优于我们的基于角色适配器和真实CrewAI（v1.14），在GPT-4o Mini和GPT-4o上优于基于角色协作——这一发现的一致性表明这是协作策略的属性，而非我们实现的人造产物。
- **单次运行评估**：每个配置每个任务仅运行一次。没有多次重复，我们无法量化LLM非确定性（即使在temperature=0下）造成的得分方差。报告的平均得分差异小于0.05应被谨慎解读。
- **V4 Pro定价**：我们使用了V4 Pro的促销定价（$0.435/$0.87）。按标准定价（$1.74/$3.48），V4 Flash的成本优势将更加显著。

### 6.3 未来工作

我们计划在以下方向扩展MAB：

1. **更多任务**：增加协商、辩论、工具协调和长期规划任务。
2. **更多框架**：随着Agno、ChatDev、MetaGPT等新框架的成熟进行集成。
3. **人类评估**：用人类判断补充自动评分，特别针对开放式任务。
4. **多模型对比**：评估框架性能如何随底层LLM的不同而变化。
5. **失败分析**：对失败模式（幻觉共识、无限循环、任务放弃）进行分类和量化。

### 6.4 给从业者的建议

基于我们的发现，我们提供以下具体指导：

1. **默认使用成本效益好的模型搭配多步协作。**V4 Flash Multi（0.91, 100%, $0.016）在成本-质量帕累托边界上支配所有其他配置。
2. **不要假设更大的模型能做出更好的智能体系统。**GPT-4o和V4 Pro——尽管每token更贵——在多步协作下不及V4 Flash。协作策略才是决定因素。
3. **对你的具体任务进行基准测试。**9个任务中有3个被所有配置轻易解决，另有几个展示了接近天花板的性能。如果你的用例也类似饱和，多智能体的开销可能是浪费。
4. **从简单开始，仅在需要时升级。**首先运行单智能体基线。仅在基线失败时添加协作——代码生成和多跳推理是最强的候选。
5. **警惕延迟陷阱。**V4 Pro Multi（164秒/任务）和GPT-4o Multi（$0.184）是被压制的替代方案。成本和延迟在配置间可以相差10倍而达到相同的成功率。

## 7. 结论

我们提出了Multi-Agent Collaboration Benchmark (MAB)——一个开源工具包，用于在标准化协作任务上比较多智能体AI系统。MAB填补了智能体工程生态系统中的一个关键空白：对能够联合考虑模型选择和协作策略的轻量级、框架无关评估的需求——这是现有基准测试工作未充分探索的维度。通过模块化适配器架构、插桩式指标采集和精心策划的跨越4个类别的9任务套件，MAB使从业者能够在架构选择上做出数据驱动的决策。

我们的实验比较了12种配置（包括一个真实的CrewAI实现），跨越4个模型类型和2个厂商，为从业者产生了可操作的洞察。**普遍协作红利**意味着添加研究员-分析师-撰稿人流水线可以提升每一个测试过的模型。**能力天花板效应**意味着增益在最需要的地方最大——弱模型面对困难任务。**简单性原则**——简单流水线优于复杂的角色扮演——意味着团队应投资于任务分解而非提示词工程。**知识边界**——协作无法填补事实性缺口——意味着在扩展智能体之前先诊断失败模式至关重要。综合来看，这些发现表明通往有效多智能体系统的路径不在于更大的模型或更精细的智能体人格，而在于将正确的协作结构与正确的任务以正确的成本匹配起来。

MAB以MIT许可证作为开源软件发布。我们邀请社区贡献新的任务、更多框架的适配器，以及解决本研究中观察到的天花板效应的更难基准实例。
